# Scenario Persistence Plan

Date: 2026-05-30
Status: design plan only; no migration applied

## Purpose

The scenario loop is now the product's new main learning path, but only the
AI-generated battle-pack cache is durable. This plan describes how to move
scenario progress, battle sessions, attempts, and reports from hardcoded/mock
client state into Supabase without changing the live database tonight.

## Current Reality

### Durable Today

- AI battle packs are generated by `/api/generate-battle-pack`, cached by a
  deterministic `cache_key`, and upserted into `public.battle_packs`
  (`app/api/generate-battle-pack/route.ts:74-90`,
  `app/api/generate-battle-pack/route.ts:227-248`).
- The applied/manual migration shape for that cache is
  `supabase/migrations/20260529_000002_battle_packs.sql`, with RLS enabled,
  authenticated select access, and writes reserved for the service role route
  (`supabase/migrations/20260529_000002_battle_packs.sql:1-30`).

### Mock Or Client-Only Today

- Scenario map progress is hardcoded in `MOCK_SCENARIO_PROGRESS`; it is not
  stored per user, language, level, or device (`lib/scenario-progress.ts:13-59`).
- Scenario battle/exam reports are `StoredBattleReport` objects written to
  `localStorage` under `clashlingo:battle-report:<sessionId>`
  (`lib/battle-session.ts:14-35`,
  `lib/battle-session.ts:127-161`).
- The scenario battle runner creates a report client-side and then routes to the
  report page (`components/BattlePage.tsx:191-204`).
- The report page reads the report back from `localStorage`
  (`components/BattleReportPage.tsx:71-100`).
- The AI opponent score is simulated in `lib/battle-scoring.ts`, not represented
  by a persisted attempt row (`lib/battle-scoring.ts:155-189`,
  `lib/battle-scoring.ts:220-282`).

## Target Data Model

The code already defines the conceptual shapes:

- `BattleSession`, `BattleAttempt`, and `BattleReport` in
  `lib/scenario-types.ts:129-189`.
- `ScenarioProgress` in `lib/scenario-progress.ts:3-6`.
- `StoredBattleReport`, which is the richer current report snapshot in
  `lib/battle-session.ts:16-35`.

The database should make those concepts durable while preserving the current
`battle_packs` cache as shared content.

### Design Principles

- Persist scenario gameplay by user, scenario, target language, and level.
- Treat `battle_packs` as shared immutable-ish content; sessions reference the
  pack cache key they ran against.
- Prefer server/service-role writes for sessions, attempts, reports, and progress
  so clients cannot forge scores.
- Allow authenticated users to read their own progress and sessions they
  participate in.
- Keep AI-only and future friend-vs-friend sessions in one schema by preserving
  `mode`, `opponent_type`, and nullable `opponent_user_id`.
- Store raw per-question evaluated results as JSONB initially so the product can
  iterate without a premature normalized question-result table.

create table if not exists public.scenario_progress (

create or replace function public.is_scenario_session_participant(

## Migration And Wiring Plan

1. **Write a real migration file in a future session.**
   - File: `supabase/migrations/<timestamp>_scenario_persistence.sql`.
   - Start from the draft above, then review naming, constraints, and whether
     realtime publication should include sessions/attempts/reports.
   - Do not rewrite `20260325_000001_baseline.sql`; `supabase/README.md`
     requires follow-up changes to be additive migrations
     (`supabase/README.md:18-30`).

2. **Have the human apply the migration manually.**
   - The repo currently does not use Supabase CLI for migrations.
   - The human should apply via Supabase SQL Editor, matching the existing
     `battle_packs` manual workflow.

3. **Add server route(s) for scenario sessions.**
   - Candidate files:
     - `app/api/scenario-sessions/route.ts` for create/read session.
     - `app/api/scenario-sessions/[id]/attempt/route.ts` for submit attempt.
     - `app/api/scenario-progress/route.ts` for reading progress summaries.
   - The routes should authenticate the user, use service role writes, and reuse:
     - `buildBattleQuestionOrder` from `lib/battle-session.ts:37-39`
     - `evaluateBattleQuestion` from `lib/battle-scoring.ts:220-282`
     - `evaluateBattleSession` from `lib/battle-session.ts:41-125`

4. **Move session creation out of client-generated mock ids.**
   - Replace the `mock-${slug}-stage-${stage}` id creation in
     `StageBriefingPage` with a server-created session
     (`components/StageBriefingPage.tsx:63-70`).
   - Replace the `mock-exam-${slug}-stage-${stage}` id creation in
     `ScenarioExamLandingPage` with a server-created solo exam session
     (`components/ScenarioExamLandingPage.tsx:29-36`).

5. **Persist attempts and reports server-side.**
   - `BattlePage` currently builds and saves a report in the browser
     (`components/BattlePage.tsx:191-204`).
   - Future wiring should POST raw answers/timing to the server. The server should
     evaluate, insert/update `scenario_battle_attempts`, produce
     `scenario_battle_reports`, and return the report id/session id.
   - Keep `localStorage` only as temporary resume/cache if needed, not as source
     of truth.

6. **Replace mock progress reads.**
   - Swap `lib/scenario-progress.ts` from `MOCK_SCENARIO_PROGRESS` to a typed
     client/server helper that reads `scenario_progress`
     (`lib/scenario-progress.ts:13-59`).
   - Update scenario map surfaces to handle loading/error/empty states rather than
     assuming synchronous mock values.

7. **Define progress rules before writing progress.**
   - Product must decide whether a stage is completed by:
     - finishing scenario exam
     - winning scenario clash
     - meeting a score threshold
     - any completed attempt
   - Until that is decided, routes can persist attempts/reports without advancing
     `scenario_progress`.

8. **Add tests and smoke coverage.**
   - Unit-level: scoring/report server route with fixed inputs.
   - Integration-level: scenario session create + dry submission against a test DB
     or mocked Supabase client.
   - Browser-level: `/scenarios` -> stage briefing -> start solo exam -> report
     happy path once persistence is in place.

## API Shape Sketch

These are draft routes, not implemented tonight.

### `POST /api/scenario-sessions`

Input:

```json
{
  "scenarioSlug": "cafe",
  "stage": 1,
  "mode": "clash",
  "opponentType": "ai"
}
```

Server behavior:

- Authenticate user.
- Resolve profile target language and default level, or accept validated explicit
  values if product wants overrides.
- Ensure a battle pack exists by reusing `/api/generate-battle-pack` logic or a
  shared server helper.
- Insert `scenario_battle_sessions` with a deterministic question order.
- Insert a `scenario_battle_attempts` row for the owner.

### `POST /api/scenario-sessions/[id]/submit`

Input:

```json
{
  "answers": {
    "cafe-s1-q1": "b",
    "cafe-s1-q2": "coffee"
  },
  "timings": {
    "cafe-s1-q1": 3,
    "cafe-s1-q2": 6
  }
}
```

Server behavior:

- Authenticate user and verify session participation.
- Load the referenced `battle_packs.pack`.
- Evaluate each answer with the shared scoring engine.
- Persist the attempt and report.
- If progress rules are approved, update `scenario_progress`.

## Open Risks

- **Progress semantics are undecided.** The schema can support progress, but the
  product must choose what counts as clearing a stage.
- **AI-only vs friend battles.** The current implemented scenario path is AI-only;
  friend PvP would need invitation/session matching rules before relying on
  `opponent_user_id`.
- **Rivalry ledger relationship.** Scenario reports should not silently update the
  existing rivalry ledger unless the product explicitly merges friend battles into
  the scenario engine.
- **Scoring trust boundary.** Current scoring is client-side. Once persisted, score
  calculation should move server-side to avoid forged scores.
- **Report shape drift.** `StoredBattleReport` is richer than `BattleReport` in
  `scenario-types`. The future implementation should either update the shared type
  or store a stable `report` JSON snapshot with a version.
- **Realtime cost/need.** Scenario AI-only reports may not need realtime. Friend
  scenario battles probably will, but the exact tables to add to
  `supabase_realtime` should wait for the PvP decision.

## Open Questions

- Is scenario progress scoped by target language and level, or should clearing
  `cafe` stage 1 count globally across all languages/levels?
- Should solo exam mode unlock stages, or is it a checkpoint while clash is the
  official completion mode?
- Should simulated AI attempts be persisted as synthetic attempt rows, or should
  AI scores live only inside reports?
- Should scenario friend battles become the successor to rivalry exams, or remain
  a separate mode?
